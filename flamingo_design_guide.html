Project Context

What FlamingoSank is and how the two Flamingo repos relate

Yos-X (枫灵剑影) maintains two connected repositories under the Flamingo brand. FlamingoSank (少女投江, literally "girl sinking") is the core shared library — its name is a poetic reference to the idea of diving deep into a problem. FlamingoHere is the user-facing Android music player application that consumes it, described as "a local music player for Android, styled after Apple Music."

📦
FlamingoSank (this repo)
Core library: custom overscroll engine (overscroll_core), shared data models, stability config. Written entirely in Kotlin. Used as a module dependency.
🎵
FlamingoHere
The full music player app. Apple Music–inspired UI, local audio playback, album art, playlists, and library navigation — built on top of FlamingoSank's engine.
🛠
Tech Stack
Kotlin 2.0, Jetpack Compose (Kotlin Compose plugin), AGP 8.4, JitPack for custom deps, Sonatype snapshots for cutting-edge libraries.
Key insight: The decision to extract the overscroll engine into its own module (overscroll_core) is itself a design principle — physics-based scroll behaviour is a first-class concern, not a CSS afterthought. Apple Music's signature feel lives almost entirely in how content responds to gesture momentum.

Architecture Principles

How the project is structured and why it matters for smooth apps

Module Separation

The project splits into at least three Gradle modules: the application (app), the physics/scroll engine (overscroll_core), and an implied shared data module. This follows the Android Single Responsibility principle at the module level — each boundary is independently compilable, testable, and replaceable.

app UI screens, navigation, ViewModels, media playback UI Layer
overscroll_core Custom physics engine: spring damping, edge stretch, momentum curves Motion Layer
yos.music.player.data.* YosMediaItem, PlayList, MusicLibrary, FolderIndex, PlayStatus Data Layer
stability_config.conf Explicit keep-rules for ProGuard/R8 — every data class serialised at runtime is named here Build Layer

Data Model Design

The stability_config.conf reveals the full domain model. Each entity has a clear single responsibility:

Core Domain Entities (from stability_config.conf) Data
YosMediaItem
Media wrapper
A single playable track: file path, metadata (title, artist, album, duration), artwork reference. Wraps Android's MediaItem with app-specific extensions like play count and last-played timestamp.
PlayList / PlayListV1
Ordered collection
An ordered list of YosMediaItem references with metadata (name, cover art, creation date). V1 suffix suggests versioned serialisation — the old format is kept for migration compatibility.
FavPlayListLibrary
Special playlist
Dedicated type for the "Favourites" collection — separated from generic playlists so the UI can treat it specially (always-visible, heart icon shortcut), mirroring Apple Music's "Loved" distinction.
MusicLibrary
Root index
Aggregates all scanned Folder objects and their tracks. Acts as a single source of truth for what's on the device, rebuilt when MediaStore changes.
PlayStatus
Playback state
Current playback state snapshot: playing/paused, current position, shuffle and repeat modes. Separated from YosMediaItem so UI can observe state changes without re-rendering track metadata.

Kotlin 2.0 + Compose Compiler Plugin

Using Kotlin 2.0 with the standalone Compose compiler plugin (org.jetbrains.kotlin.plugin.compose) rather than the bundled Compose Kotlin extension is a deliberate performance choice. The new compiler produces smaller, faster-skipping composables and enables Strong Skip Mode — meaning unchanged composables are completely skipped during recomposition, not just read-checked.

03

The Custom Overscroll Engine

The most distinctive technical decision in the project — why and how

The existence of a standalone overscroll_core module is the single most revealing architectural choice in FlamingoSank. It tells you exactly what the developer considered non-negotiable: the physics of how lists feel when you reach the end.

Why Android needs this: Android 12 introduced a "stretch" overscroll (EdgeEffect). But it doesn't match iOS's rubber-band feel — iOS uses a spring with variable resistance, while Android's default is a one-size-fits-all stretch. FlamingoSank replaces Android's EdgeEffect entirely with a custom spring simulation.

Principles of a Physics-Correct Overscroll

🌊
Variable Resistance
The further the content is pulled past the edge, the more resistance is applied. This is a non-linear curve — small pulls feel almost free; large pulls feel increasingly sticky, preventing users from "launching" content off-screen.
🔁
Spring Return
On release, content snaps back with damping — it overshoots slightly and settles, rather than teleporting back. Damping ratio controls whether the bounce feels firm (high damping) or bouncy (low damping).
⚡
Velocity Inheritance
The spring initial velocity is set to the gesture's fling velocity at the moment of edge contact. Fast flings produce energetic bounces; slow pushes produce gentle snaps. This is what makes it feel "alive."
📐
Content Transformation
Rather than moving a glow or edge decoration, the actual list content translates and optionally scales slightly. This is the iOS approach — the content itself appears to stretch, not a separate effect layer.

Implementation Pattern in Jetpack Compose

// overscroll_core module — core principle class SpringOverscrollEffect( private val coroutineScope: CoroutineScope ) : OverscrollEffect { private val springSpec = SpringSpec( dampingRatio = 0.65f, // ~Apple Music feel stiffness = 200f ) private val offset = Animatable(0f) override fun applyToScroll( delta: Offset, source: NestedScrollSource, performScroll: (Offset) -> Offset ): Offset { val consumed = performScroll(delta) val leftover = delta - consumed // Apply rubber-band resistance to leftover val resistance = rubberBand(leftover.y, maxPull = 200f) coroutineScope.launch { offset.snapTo(offset.value + resistance) } return delta // consume all so host doesn't also move } private fun rubberBand(delta: Float, maxPull: Float): Float { // Non-linear decay: feels free at start, heavy at end return delta * (1f - (kotlin.math.abs(offset.value) / maxPull).coerceIn(0f, 1f)) } }

Do and Don't

✓ DO

Move the actual content using graphicsLayer { translationY = overscrollOffset } on your list container. Tie spring stiffness and damping to your visual density — denser lists need stiffer springs.

✗ DON'T

Override EdgeEffect via reflection or try to intercept Android's built-in overscroll glow. Disable it completely with overScrollMode = OVER_SCROLL_NEVER and replace it entirely.

Motion & Animation

How Flamingo achieves Apple-quality transitions on Android

Apple Music's UI is defined by its transitions as much as its layouts. Every screen change, mini-player expansion, and track switch has a coherent choreography. Flamingo achieves this through three motion categories:

Category 1 — Shared Element Transitions Motion

Album artwork morphs from a thumbnail in the list to a full-screen hero when the player expands. In Compose, this is achieved with SharedTransitionLayout (Compose 1.7+) combined with sharedElement() modifier. The key principle: the element never disappears — it moves. No fade-in/out, no cross-dissolve. The artwork rect animates directly from origin to destination using a spring.

// In the track list item AsyncImage( modifier = Modifier .sharedElement( state = rememberSharedContentState("artwork-${track.id}"), animatedVisibilityScope = animatedScope ) .clip(RoundedCornerShape(8.dp)) )
Category 2 — Mini-Player Expansion Motion

The persistent mini-player at the bottom expands to a full player screen. This is not a navigation transition — it's a layout animation. The mini-player's height animates from ~60dp to full screen. Content fades in as the card expands. Use animateDpAsState with a spring for height, and AnimatedVisibility for the detailed controls that appear only in expanded state.

Principle: the two states share a single composable with animated properties, not two separate composables with a navigation transition between them. This preserves the visual continuity that makes it feel like one thing transforming, not two screens replacing each other.

Category 3 — List & Scroll Choreography Motion

Album grid items scale slightly on press (scale(0.97f)), header art blurs and fades as it scrolls behind the collapsed toolbar (using graphicsLayer tied to scroll position), and the Now Playing bar fades in with a gentle spring when playback starts.

// Header parallax + blur on scroll Box( modifier = Modifier.graphicsLayer { val scrollFraction = (scrollOffset / maxHeaderHeight).coerceIn(0f, 1f) alpha = 1f - scrollFraction translationY = scrollOffset * 0.4f // parallax renderEffect = BlurEffect( radiusX = scrollFraction * 12f, radiusY = scrollFraction * 12f ) } )

Spring Configuration Reference

Overscroll bounce dampingRatio=0.65, stiffness=200 — bouncy but controlled High energy
Player expand/collapse dampingRatio=0.80, stiffness=300 — firm, confident Structural
Shared element (artwork) dampingRatio=0.9, stiffness=400 — fast, no bounce Precise
Item press feedback tween(80ms) down, spring(stiffness=500) up — snappy Micro
05

UI Component Patterns

The specific components Apple Music uses and how to build them faithfully

Album Art with Dynamic Colour Extraction UI

Apple Music's player background adapts to the album art's dominant colour. This is achieved with Android's Palette API. Load the bitmap, extract the dominant swatch, then animate the background colour using animateColorAsState with a tween when the track changes.

val dominantColor by produceState<Color>(initialValue = Color.DarkGray) { val bitmap = loadBitmap(track.artworkUri) val swatch = Palette.from(bitmap).generate() ?.dominantSwatch value = swatch?.let { Color(it.rgb) } ?: Color.DarkGray } val animatedBg by animateColorAsState( targetValue = dominantColor, animationSpec = tween(durationMillis = 600) )
Blurred Background (Frosted Glass) UI

The Apple Music mini-player and some overlays use a frosted/blurred glass effect behind them. On Android, this is done with RenderEffect.createBlurEffect() (API 31+) applied via graphicsLayer. For older APIs, pre-blur a scaled-down copy of the background bitmap. The Flamingo approach: maintain a BlurTransformation in Coil's image pipeline to produce a blurred artwork version for the player background.

Track List Item UI

Each track row contains: 44dp square artwork thumbnail (8dp rounded corners), title in 15sp/500 weight, artist in 13sp/400 weight muted, duration right-aligned, and a "more" button. Key principle: no dividers between rows — breathing room comes from padding (14dp vertical), not lines. The currently playing track shows an animated waveform indicator instead of the artwork.

✓ DO

Use LazyColumn with key = { track.id } so Compose animates insertions/removals automatically. Let Coil handle artwork caching — it handles memory pressure gracefully.

✗ DON'T

Load full-resolution artwork into list thumbnails. Use a 64px Coil target size. The full image is only needed in the player view.

Persistent Mini-Player Bar UI

Positioned above the bottom navigation bar using a Scaffold with custom bottomBar. The mini-player is always composed (not conditionally shown) to preserve media session state. Visibility is controlled by AnimatedVisibility with a slide-up enter transition. It contains: 44dp artwork, track title (marquee scrolling when overflowing), play/pause button, and a tap target that triggers the full player expansion.

Seek Bar / Progress Indicator UI

Apple Music's seek bar has a larger thumb that appears only while dragging. Build with a custom Slider composable: track at 4dp height, thumb invisible at rest (zero size), animating to 20dp diameter on drag. The hit target is always 44dp regardless of visual size. Position timestamps update every 500ms via a LaunchedEffect coroutine loop — not every frame — to avoid recomposing the entire player screen at 60fps.

Section Headers (Library Navigation) UI

The library tab uses sticky section headers (Albums, Artists, Playlists, Songs) implemented with LazyListScope.stickyHeader. Each header has a translucent background using surface.copy(alpha = 0.92f) so content is visibly beneath it. The sort button in the top-right uses a DropdownMenu with the same translucency treatment.

Data Layer Design

How media data flows from device storage to the UI

YosStringWrapper — Why Wrap Strings?

The data model includes a YosStringWrapper class. This pattern — wrapping a primitive in a value class — is used for type safety and to prevent passing the wrong string to the wrong field. An artist name and a file path are both strings, but they're not interchangeable. Kotlin value classes (@JvmInline value class ArtistName(val value: String)) achieve this at zero runtime cost.

MediaStore Scanning

The MusicLibrary entity is populated by querying Android's MediaStore. The scan should happen on a background dispatcher and only when MediaStore signals a change (via ContentObserver), not on every app start. The result is cached in a StateFlow in a ViewModel with a longer lifecycle than the screen (use hiltViewModel() at the navigation host level).

// Repository pattern — single scan, broadcast to all consumers class MusicRepository @Inject constructor( private val context: Context, private val scanner: MediaStoreScanner ) { private val _library = MutableStateFlow<MusicLibrary>(MusicLibrary.Empty) val library: StateFlow<MusicLibrary> = _library.asStateFlow() suspend fun refresh() { _library.emit(scanner.scan()) } }

PlayStatus — Reactive Playback State

Because PlayStatus is a separate class from the media item, the UI can efficiently observe only what changed. If the track changes, the artwork composable recomposes. If only the position changes, only the seek bar recomposes. Achieving this requires using derivedStateOf in Compose to subscribe to specific fields rather than the entire state object.

Performance Engineering

Staying at 60/120fps across the board

🎨
R8 Full Mode + stability_config
The stability_config.conf tells R8 which classes are stable for optimisation. Without it, R8 would treat Kotlin data classes as potentially unstable and skip dead-code elimination. Naming all data model packages here enables aggressive minification and inlining.
🏗
Compose Stability
Every composable that touches media state should take stable parameters. Mark data classes with @Stable or @Immutable where appropriate. Use the Compose compiler metrics flag to verify no unexpected recompositions are occurring.
🔗
Lazy Loading with Keys
All lists use key = { item.id }. This lets Compose track items through changes: an item moved in the list triggers a positional animation, not a full redraw. The overscroll engine relies on this — it works at the list container level, not per-item.
🧵
Background Coroutines
All file I/O (MediaStore queries, artwork loading, playlist serialisation) runs on Dispatchers.IO. All computation (sorting, filtering, Palette extraction) runs on Dispatchers.Default. The main thread only receives final StateFlow emissions.

Image Loading Strategy (Coil)

Album artwork is the most memory-intensive asset. The Flamingo approach:

  • List thumbnails: 64×64dp target size, memory cache enabled, no disk cache (regenerated from MediaStore)
  • Player full art: full resolution, disk-cached, preloaded when the mini-player first shows
  • Blurred background: generated from the full art via Coil's BlurTransformation, separate cache key
  • Placeholder: a solid colour from dominantColor extraction — no generic grey box

Optimisation Checklist

Use this when building or reviewing an Apple Music-style Android app

Architecture

  • ✓Overscroll/physics logic is in its own module — not inside Activity or Fragment
  • ✓Data model entities each have a single responsibility (media item ≠ playlist ≠ library ≠ play status)
  • ✓All domain classes are named in stability_config.conf or equivalent keep-rules for R8
  • ✓Kotlin 2.0 + standalone Compose compiler plugin used (not bundled extension)

Motion & Feel

  • ✓System overscroll disabled; replaced with custom spring-based OverscrollEffect
  • ✓Album artwork uses sharedElement() for list→player transition
  • ✓Mini-player expands via layout animation, not a navigation transition
  • ✓Spring configs are varied per interaction type (bouncy overscroll ≠ firm structural open)
  • ✓Press feedback on interactive items: slight scale down on press, spring back on release
  • ✓Header parallax tied to scroll position via graphicsLayer { translationY }

UI & Visual

  • ✓Dynamic background colour extracted from album art via Palette API, animated with tween
  • ✓Frosted blur on player overlays using RenderEffect.createBlurEffect()
  • ✓Seek bar thumb invisible at rest, appears only on drag — large hit target always present
  • ✓No row dividers — spacing communicates grouping
  • ✓Mini-player bar always composed; AnimatedVisibility controls opacity/position only
  • ✓Currently playing track shows animated waveform, not static artwork

Performance

  • ✓All lists have key = { item.id } lambda
  • ✓Data classes annotated with @Stable or @Immutable where applicable
  • ✓Seek bar position updates at 500ms interval — not every frame
  • ✓Thumbnail target size capped at 64dp × 64dp in Coil requests
  • ✓MediaStore scan on Dispatchers.IO, result emitted to StateFlow
  • ✓Compose compiler metrics checked — zero unexpected recompositions in list scroll
